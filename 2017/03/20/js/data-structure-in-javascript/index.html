【持续更新】JavaScript中的数据结构

发表于 2017-03-20 | 分类于 算法 |

数组

在 js 中，除了基础类型 Boolean、Number、String、Null和Undefined以外，其他都是对象。

所以在这里简单记录的是，作为一种基础数据结构，js中的数组有什么特殊的地方，以及在性能上需要注意的地方。

我们都知道数组这种数据结构，它的经典实现就是：内存中一段连续的位置分配来存储数据的线性集合。所以在C等语言中，它是需要在声明时指定长度的，以便在栈区分配内存，并且长度是不可变的。

不过 js 的数组最突出的特点就是：长度可变。

那么，会不会对数组本身的优点即“快速读取”带来影响呢？

在V8中，数组的存储方式基本上就是：

  1. Fast Elements 传统的线性存储方式，动态分配内存
  2. Fast Holey Elements 如果出现了不连续的索引位置赋值，则转换为Fast Holey Elements，没有赋值的位置被赋值undefined，动态分配内存
  3. Dictionary Elements 以 Dictionary 本质是 HashTable的方式存储，内存或许上节省了，但是使用的开销增加了

所以，Fast Elements最接近原本数组的特性，而其他的两种都是由它“转换”而来的，在性能上有所下降——基本都是在赋值的时候。

需要注意的是，如果 key 不是无符号整数，这时已经不按数组处理了，而是按照Object来处理的，此时不会去处理 length 属性。

性能注意点

下面的几个性能点可能都要执行量过10W甚至100W的时候才能看出明显区别……

使用非负整数作为索引下标，从0开始用

使用非整数、字符串、负数这样的索引下标，实际上会把它作为一般的对象key来处理了，作为属性来使用，其实要比常规数组慢很多！

如果不从0开始用，那直接就带洞了，并且还可能变成Dictionary Elements，详看下面。

最好是避免洞的出现

所谓的洞，就是数组中存在元素未赋值，也就是undefined。

new Array(1000)这般的预分配的数组，如果长度小于100000（in v8），这时候其实倒是无所谓的，底层的实现还是C数组的模式，也就匹配了上面的Fast Holey Elements而已，多出了一点开销在“洞”的处理上。

来看不是预分配的数组

如果是连续使用索引，从0开始一直到lengt - 1，无洞，那么其实就会一直保证是C数组模式了，效率赞赞的

如果跳着用，而要设置的索引值超出数组当前长度已经达到kMaxGap决定的阈值——通常是1024——这时就会被转换为Dictionary Elements了，效率下降很多

错误的使用可能会导致不必要的内存消耗，以及潜在的转换消耗。

从上面的例子可以看出，提前分配内存，就是空间换时间，不然如果被转成Dictionary Elements就是时间换空间了。

其实在浏览器端的话，时间更重要，因为在浏览器端很少去处理超大规模的数据，反倒是 Node 里可能要更注意一点空间的处理。

所以如果是较小的数组使用，例如已经知道最多用100这么大的数组，又有可能需要有洞，那么提前分配为Array(100)不失为一个好主意。

但是如果是长度超过64000（PagesPerChunk的大小限制？）这种，使用时的重点就在于：1. 动态分配内存 2. 连续使用索引，总之是避免一次初始化过大的数组，更要避免Dictionary Elements的转换。

Dictionary Elements可能会被v8的启发式算法弄回C数组模式

不过，还是别指望了，这个只是个优化而已，本质是检查洞数量是否够少。

元素值的类型

尽量使用一样的类型，如果不行，那最好是使用字面量直接初始化，避免一堆隐式的封箱处理(boxing)。

数组也是对象，而在v8中有这样一个优化：首先将key为非负整数的元素成为Element，对于Element有一个优化就是如果存储的值都是>=0的整数（符合32位的条件），那么v8就认为存储的东西都是Fast small integers，以每个都是带符号31位整数去连续的分配内存，即C数组模式。

而如果这时加了个小数、超大整数或者-0这样的数，就会导致将数组的元素类型提升为Fast doubles，从32位变64位了，好么，重新申请内存，再复制当前的值过去。

其实这时候还是很快的，因为所有的数字都是无封箱存储的。

若再扔个字符串或对象，又转为Fast values类型，这时候就很慢了。

总结

大体如此罢，随时补充，反正如果能确保Dictionary Elements别冒出来，js数组的性能还是很不错的。

对象

对象在最差的时候就是 Dictionary ，不过其实是可以优化为连续内存模式的，这个在上面的部分已经提到了，只要key是满足非负整数的元素，就可能会触发相应的优化。

Element与普通属性是分开的，性能上其实互不影响！所以，Object的数组化使用，也可能会是个优化点……

来看另外一个v8的优化：隐藏类。

隐藏类

js的对象可以动态的修改属性，这导致了其结构可能会频繁改变，这样字典类型存储属性的话，每次都要动态的去定位属性对应的下标地址，然后访问。

为了提升读取性能，v8是利用动态创建隐藏内部类的方式动态地将属性的内存地址记录在对象内，从而提升整体的属性访问速度。

每次修改，都是创建一个新的隐藏类，然后尽量的复用之前的隐藏类的已有属性（一次一个属性/方法）。

而在new的时候，其实不会再创建了，而是使用已有的，除非又去添加新的属性。

这就带来了一个优化处理：尽量在构造函数内去连续的声明完，不要在new出来新实例后，又手动的增加新的属性。

Dictionary Mode（for复杂属性的对象）

其实至少也能当个 Dictionary 来用，还是小优化过的…… 说是HashTable也OK。

不过这种模式下，性能最差了…… 尽管存储开销要小于上面的隐藏类，但是初始化时，哈希表中的所有的键与值都被设置为了undefined，当插入新的数据时，会去计算键名的哈希值的低位会被当做初始的存储索引地址。如果此地址已经被占用了，V8 会尝试向下一个地址进行插入，直到插入成功。

每次都要计算，所以肯定是个不大好消耗。

总之，js中的对象起到的作用已经超过了基础的数据结构 Dictionary 和 HashTable，也不必费心去实现了。
